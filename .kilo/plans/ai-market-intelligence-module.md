Version: 0.1
Status: Draft
Document Type: Architecture / AI / Market Intelligence

## 0. Current Architecture Review

APEX is designed as a modular monolith that is microservices-ready. The current architecture already defines Market Intelligence, News Intelligence, Sentiment Intelligence, and AI agents as future modules. The current backend has market collection for candles, order book snapshots, funding rates, and open interest, but it does not yet include news collection, sentiment analysis, fear and greed scoring, or market regime detection.

Relevant existing references:

- `docs/03_SYSTEM_ARCHITECTURE.md` defines Market Intelligence, News Intelligence, Sentiment Intelligence, and AI agents.
- `docs/09_AI_ARCHITECTURE.md` defines Stage 2 Market Intelligence with Market Analyzer, News Analyzer, Sentiment Analyzer, Macro Analyzer, and Whale Analyzer.
- `docs/04_DOMAIN_MODEL.md` already declares NewsArticle, NewsSource, SentimentRecord, MacroEvent, and WhaleActivity.
- `docs/05_DATABASE_ARCHITECTURE_PART1.md` reserves future database schemas for `intelligence`, `ai`, `research`, and `knowledge`.
- `docs/08_WEB_ARCHITECTURE.md` already plans News Intelligence, Sentiment Intelligence, AI Intelligence, and real-time updates.
- `backend/app/events/types.py` defines the base `ApexEvent` shape used by the event bus.
- `backend/app/tasks/collectors/market_collector.py` already collects market candles through Celery.

Target principle:

- Start with deterministic, explainable intelligence scoring.
- Keep LLM usage optional and bounded to summarization, classification assistance, and research explanation.
- Do not allow intelligence outputs to execute trades directly.
- Route intelligence outputs to Strategy, Risk, Portfolio, Notification, and AI agents through validated events.

## 1. Architecture

### 1.1 High-Level Architecture

```text
External Data Sources
├── News APIs / RSS / Web Feeds
├── Social Platforms
├── Market Data APIs
└── Macro / Economic Calendars

        ↓

Collector Layer
├── News Collector
├── Social Sentiment Collector
├── Market Feature Collector
└── Source Health Monitor

        ↓

Normalization Layer
├── Source Normalizer
├── Asset / Symbol Resolver
├── Entity Linking
├── Deduplication
└── Data Freshness Validator

        ↓

Analysis Layer
├── News Classification
├── Event Detection
├── Social Sentiment Analysis
├── Market Sentiment Aggregation
├── Fear and Greed Index
└── Market Regime Detection

        ↓

Intelligence Aggregation Layer
├── Component Scoring
├── Confidence Scoring
├── Intelligence Snapshot Builder
├── Alert Detection
└── Explanation Builder

        ↓

APEX Core Integration
├── Event Bus
├── PostgreSQL Intelligence Store
├── Redis Intelligence Cache
├── Strategy Engine

├── Notification Service
└── AI / Research Agents

        ↓

API and UI
├── News API
├── Sentiment API
├── Market Regime API
├── Intelligence API
├── WebSocket Updates
└── Dashboard Pages
```

### 1.2 Layer Responsibilities

| Layer | Responsibility |
|---|---|
| Collector Layer | Pulls raw data from approved external sources with rate limits, retries, and source health tracking. |
| Normalization Layer | Converts raw data into APEX-standard entities, resolves symbols/assets, removes duplicates, and validates freshness. |
| Analysis Layer | Produces classified news, detected events, sentiment scores, fear and greed values, and market regime labels. |
| Aggregation Layer | Combines component scores into per-asset and per-market intelligence snapshots. |
| Integration Layer | Publishes events, persists outputs, caches latest snapshots, and exposes outputs to Strategy, Risk, Portfolio, Notifications, and AI agents. |
| API/UI Layer | Provides REST and WebSocket access for dashboards, Telegram, and future AI agents. |

### 1.3 Operating Modes

| Mode | Purpose |
|---|---|
| Development | Synthetic or manually seeded news, sentiment, and market data. No external paid sources required. |
| Staging | Real external sources where allowed, but no live trading impact. |
| Production | Approved sources only, strict rate limits, audited outputs, stale-data safeguards, and risk-gated consumption. |

### 1.4 Intelligence Score Model

The module should produce component scores and a composite score.

| Component | Range | Purpose |
|---|---:|---|
| News Impact Score | 0-100 | Measures expected market impact from classified news and detected events. |
| Social Sentiment Score | 0-100 | Measures public opinion direction and intensity. |
| Market Sentiment Score | 0-100 | Measures broader market tone using price, volume, liquidity, and social context. |
| Fear and Greed Index | 0-100 | Measures market fear versus greed, where low means fear and high means greed. |
| Regime Confidence | 0-100 | Measures confidence in the detected market regime. |
| Intelligence Score | 0-100 | Composite score used by dashboards, agents, and strategy features. |

Initial score direction:

- 0-20: Extreme fear / crisis / severe negative pressure.
- 21-40: Fear / risk-off.
- 41-60: Neutral / mixed.
- 61-80: Greed / risk-on.
- 81-100: Extreme greed / overheated conditions.

The score is an intelligence feature, not a trade signal.

## 2. Components

### 2.1 News Intelligence Components

| Component | Responsibility |
|---|---|
| `NewsSourceRegistry` | Tracks approved news sources, reliability scores, categories, rate limits, and enabled status. |
| `NewsCollector` | Fetches articles from RSS, API, or webhook sources with retries and source health tracking. |
| `NewsNormalizer` | Converts raw articles into APEX-standard title, summary, source, URL, published time, category, and raw payload. |
| `AssetSymbolResolver` | Maps article mentions to APEX assets and market pairs. |
| `NewsDeduplicator` | Prevents duplicate articles using URL hash, title hash, and source fingerprint. |
| `NewsClassifier` | Classifies articles by category, asset relevance, market impact, and sentiment polarity. |
| `EventDetector` | Detects market-moving events such as listings, delistings, hacks, regulation, macro releases, outages, ETF approvals, rate decisions, and earnings. |
| `NewsImpactScorer` | Produces impact score from source reliability, event severity, asset relevance, recency, and market context. |
| `NewsAlertEvaluator` | Emits alerts for breaking, high-impact, or crisis-related news. |

### 2.2 Sentiment Intelligence Components

| Component | Responsibility |
|---|---|
| `SocialSentimentSourceRegistry` | Tracks social platforms, allowed keywords, symbols, rate limits, and reliability scores. |
| `SocialSentimentCollector` | Collects public posts, mentions, comments, or feeds from approved sources. |
| `SocialTextNormalizer` | Normalizes text, emojis, hashtags, cashtags, URLs, and platform-specific metadata. |
| `SocialSentimentAnalyzer` | Produces sentiment polarity, subjectivity, confidence, and asset relevance. |
| `MentionAggregator` | Aggregates mention count, velocity, unique authors, and platform distribution. |
| `MarketSentimentAggregator` | Combines social sentiment, price action, volume, liquidity, and news impact into market sentiment. |
| `FearGreedIndexBuilder` | Builds a fear and greed value from volatility, trend, volume, breadth, sentiment, and news stress. |
| `SentimentFreshnessValidator` | Rejects stale sentiment snapshots and marks data quality status. |

### 2.3 Market Regime Detection Components

| Component | Responsibility |
|---|---|
| `MarketFeatureBuilder` | Builds features from candles, volume, order book, funding, open interest, and volatility. |
| `TrendStrengthAnalyzer` | Detects trending conditions using slope, moving averages, ADX-like trend strength, and higher-high/lower-low structure. |
| `SidewaysMarketAnalyzer` | Detects range-bound conditions using volatility compression, weak slope, and support/resistance behavior. |
| `VolatilityAnalyzer` | Detects volatile conditions using realized volatility, ATR-like movement, volume expansion, and spread changes. |
| `CrisisDetector` | Detects crisis conditions using sharp drawdowns, abnormal volume, negative news, fear score, liquidity stress, and correlated asset weakness. |
| `RegimeClassifier` | Classifies the market regime as Trending, Sideways, Volatile, or Crisis. |
| `RegimePersistenceTracker` | Tracks regime duration, transitions, and stability to reduce noisy flips. |

### 2.4 Intelligence Aggregation Components

| Component | Responsibility |
|---|---|
| `IntelligenceScoreEngine` | Combines news, sentiment, fear/greed, and regime outputs into a composite intelligence score. |
| `ConfidenceEngine` | Calculates confidence from data freshness, source count, source reliability, classifier confidence, and market data quality. |
| `ExplanationBuilder` | Produces human-readable explanations for dashboards, Telegram, and AI agents. |
| `IntelligenceSnapshotBuilder` | Creates time-windowed snapshots per asset, market pair, and market-wide view. |
| `AlertPolicyEngine` | Determines when intelligence changes require alerts or notifications. |
| `IntelligenceRepository` | Persists normalized intelligence artifacts and snapshots. |
| `IntelligenceEventPublisher` | Publishes intelligence events to the APEX event bus. |

### 2.5 Integration Components

| Component | Responsibility |
|---|---|
| `Strategy Intelligence Adapter` | Exposes regime, news, and sentiment features to strategy plugins. |
| `Risk Intelligence Adapter` | Exposes crisis, volatility, and sentiment stress features to risk checks. |
| `Portfolio Intelligence Adapter` | Exposes market-wide regime and sentiment changes for allocation context. |
| `Notification Intelligence Adapter` | Sends high-impact news, crisis alerts, and sentiment shifts to users. |
| `AI Research Adapter` | Provides structured intelligence artifacts to Research and Market agents. |

## 3. Data Flow

### 3.1 News Data Flow

```text
News Source
  ↓
News Collector
  ↓
Raw Article
  ↓
News Normalizer
  ↓
Deduplication
  ↓
Asset / Symbol Resolution
  ↓
News Classifier
  ↓
Event Detector
  ↓
Impact Scorer
  ↓
Persist NewsArticle, NewsEvent, Source Metrics
  ↓
Publish NEWS.COLLECTED, NEWS.CLASSIFIED, NEWS.EVENT_DETECTED
  ↓
Strategy / Risk / Portfolio / Notification / AI Agents
```

News flow requirements:

- Each article must have a deduplication key.
- Each article must have source reliability and data freshness metadata.
- Each article must be classified before it can affect scores.
- High-impact events must publish an alert event.
- Raw payloads must be stored in JSONB or future object storage, not as unstructured blobs only.

### 3.2 Social Sentiment Data Flow

```text
Social Source
  ↓
Social Sentiment Collector
  ↓
Text Normalizer
  ↓
Asset / Cashtag Resolver
  ↓
Sentiment Analyzer
  ↓
Confidence Filter
  ↓
Mention Aggregator
  ↓
Social Sentiment Record
  ↓
Publish SENTIMENT.SOCIAL_UPDATED
  ↓
Market Sentiment Aggregator
  ↓
Fear and Greed Index Builder
  ↓
Publish SENTIMENT.MARKET_UPDATED and MARKET.FEAR_GREED_UPDATED
```

Sentiment flow requirements:

- Sentiment records must include platform, asset, text hash, score, confidence, and timestamp.
- Low-confidence records may be stored but excluded from score aggregation.
- Aggregation must be time-windowed.
- Aggregation must include mention volume and velocity, not just polarity.
- Social sentiment must never be treated as a standalone trade signal.

### 3.3 Market Sentiment Data Flow

```text
Candles
Order Book Snapshots
Funding Rates
Open Interest
News Impact
Social Sentiment
  ↓
Market Feature Builder
  ↓
Market Sentiment Aggregator
  ↓
Fear and Greed Index Builder
  ↓
Market Sentiment Snapshot
  ↓
Publish MARKET.SENTIMENT_UPDATED
```

Market sentiment inputs:

- Price trend
- Volume trend
- Volatility
- Liquidity
- Spread
- Funding rate
- Open interest
- News impact
- Social sentiment
- Breadth across tracked assets

### 3.4 Market Regime Data Flow

```text
Market Data
  ↓
Feature Builder
  ↓
Trend Analyzer
Sideways Analyzer
Volatility Analyzer
Crisis Detector
  ↓
Regime Classifier
  ↓
Regime Persistence Tracker
  ↓
Market Regime Snapshot
  ↓
Publish MARKET.REGIME_CHANGED
```

Regime labels:

| Regime | Description |
|---|---|
| Trending | Persistent directional movement with trend confirmation. |
| Sideways | Range-bound movement with weak directional strength. |
| Volatile | High movement, wide ranges, or abnormal volume without necessarily being directional. |
| Crisis | Severe stress, sharp drawdown, liquidity stress, correlated weakness, or extreme negative intelligence. |

Regime transition rules:

- Crisis overrides all other regimes when crisis confidence exceeds threshold.
- Volatile can coexist with Trending but the persisted primary regime must remain one of the four labels.
- Sideways requires low trend strength and controlled volatility.
- Regime changes must be stabilized to avoid rapid oscillation.

### 3.5 Intelligence Snapshot Data Flow

```text
News Impact Score
Social Sentiment Score
Market Sentiment Score
Fear and Greed Index
Regime Classification
  ↓
Intelligence Score Engine
  ↓
Confidence Engine
  ↓
Explanation Builder
  ↓
Intelligence Snapshot
  ↓
API Cache, Dashboard, Event Bus, Strategy/Risk/Portfolio Adapters
```

Snapshot scope:

- Asset-level snapshot
- Market-pair-level snapshot
- Exchange-level snapshot
- Market-wide snapshot
- Time-windowed snapshot

| Raw social text | Short retention, then hash-only retention for privacy and volume control. |

| Normalized news articles | Long retention for research and audit. |
| Sentiment scores | Long retention for model training and historical analysis. |
| Fear/greed snapshots | Long retention. |
| Regime snapshots | Long retention. |
| Intelligence snapshots | Long retention, with monthly or yearly partitioning as volume grows. |

### 4.6 Database Scaling Considerations

- Use PostgreSQL partitioning for high-volume time-series tables.
- Store raw payloads in JSONB initially; move large raw payloads to object storage later.
- Cache latest snapshots in Redis.
- Use materialized summaries for dashboard widgets.
- Avoid storing raw API responses in multiple tables.
- Use idempotency hashes to prevent duplicate ingestion.
- Keep normalized scores queryable and compact.

## 5. Event Flow

### 5.1 Event Types

| Event Type | Payload Focus | Trigger |
|---|---|---|
| `NEWS.COLLECTED` | Source, article ID, asset, freshness | News collector stores normalized article. |
| `NEWS.CLASSIFIED` | Article ID, category, sentiment, confidence | News classifier completes classification. |
| `NEWS.EVENT_DETECTED` | Event type, severity, impact score, asset | Event detector identifies market-moving event. |
| `NEWS.ALERT_CREATED` | Alert ID, severity, title, message | High-impact news requires notification. |
| `SENTIMENT.SOCIAL_UPDATED` | Platform, asset, sentiment score, mention weight | Social sentiment record is analyzed. |
| `SENTIMENT.MARKET_UPDATED` | Asset, market sentiment, confidence | Market sentiment aggregation completes. |
| `MARKET.FEAR_GREED_UPDATED` | Scope, index value, label, components | Fear and greed snapshot is generated. |
| `MARKET.REGIME_CHANGED` | Market pair, previous regime, new regime, confidence | Regime classifier detects transition. |
| `INTELLIGENCE.SNAPSHOT_UPDATED` | Asset, scores, regime, confidence, explanation | Intelligence snapshot is generated. |
| `INTELLIGENCE.ALERT_CREATED` | Alert type, severity, trigger snapshot | Intelligence alert policy triggers. |
| `INTELLIGENCE.DATA_STALE` | Scope, source, last update, staleness duration | Freshness validation fails. |

### 5.2 Consumers

| Consumer | Consumes | Behavior |
|---|---|---|
| Strategy Engine | `MARKET.REGIME_CHANGED`, `INTELLIGENCE.SNAPSHOT_UPDATED` | Uses intelligence as optional features for signal generation. |
| Risk Engine | `MARKET.REGIME_CHANGED`, `INTELLIGENCE.ALERT_CREATED` | Tightens or blocks trading during crisis or volatile regimes. |
| Portfolio Engine | `INTELLIGENCE.SNAPSHOT_UPDATED` | Uses market context for allocation review. |
| Notification Service | `NEWS.ALERT_CREATED`, `INTELLIGENCE.ALERT_CREATED` | Sends alerts to web and Telegram. |
| AI Research Agent | News, sentiment, regime, and intelligence events | Stores research context and generates explanations. |
| AI Market Agent | Regime and market sentiment events | Produces market state summaries. |
| Dashboard | Snapshot and alert events | Updates UI through WebSocket or polling. |

### 5.3 Event Handling Requirements

- Events must include `correlation_id`.
- Events must include `source`.
- Events must include timestamps in UTC.
- Handlers must be idempotent.
- Failed events must move to the dead-letter queue.
- Crisis and kill-switch-adjacent events must have higher priority.
- Intelligence events must not directly create orders.
- Risk and governance policies must validate any downstream action.

### 5.4 Event Flow Example: Crisis Detection

```text
Market Regime Detector
  detects crisis regime
  ↓
MARKET.REGIME_CHANGED
  ↓
Risk Engine
  tightens trading limits
  ↓
Notification Service
  sends crisis alert
  ↓
Intelligence Snapshot Builder
  updates intelligence snapshot
  ↓
INTELLIGENCE.SNAPSHOT_UPDATED
  ↓
Strategy Engine
  may reduce or suppress signals
```

## 7. Scalability Considerations

### 7.1 Collection Scalability

- Collectors should be source-based workers.
- Each source should have independent rate limits and retry policies.
- Source health should be tracked per collector.
- Paid or low-rate-limit sources should be prioritized.
- Deduplication should happen before expensive classification.
- Collectors should support batch ingestion.
- Collection failures should not block other sources.

### 7.2 Analysis Scalability

- News classification should be parallelized by article batch.
- Sentiment analysis should be parallelized by platform and asset.
- Regime detection should be parallelized by market pair and timeframe.
- Heavy analysis should run through Celery workers.
- Deterministic classifiers should run before optional LLM classification.
- LLM calls should be rate-limited, cached, and used only for approved tasks.

### 7.3 Storage Scalability

- Use PostgreSQL partitioning for high-volume time-series data.
- Store normalized scores as compact numeric columns.
- Store raw payloads in JSONB initially and object storage later.
- Use Redis for latest snapshot cache.
- Use materialized summaries for dashboards.
- Archive old raw text and raw payloads according to retention policy.
- Use idempotency hashes to avoid duplicate records.

### 7.4 Event Bus Scalability

- Partition events by asset, market pair, or source.
- Use priority handling for crisis, news alerts, and stale-data alerts.
- Use dead-letter queues for failed intelligence handlers.
- Use bounded retries for collectors and analyzers.
- Fail fast for risk-related intelligence alerts.
- Maintain correlation IDs across news, sentiment, regime, and intelligence events.

### 7.5 API Scalability

- Latest intelligence snapshots should be served from Redis or cached query results.
- Historical queries should use indexed time ranges.
- WebSocket updates should use compact messages.
- Admin refresh endpoints should be rate-limited.
- Dashboard pages should request aggregated snapshots, not raw event streams.
- API responses must include freshness and confidence metadata.

### 7.6 Reliability and Failure Handling

| Failure Scenario | Required Behavior |
|---|---|
| News source unavailable | Mark source unhealthy, retry with backoff, continue other sources. |
| Sentiment source unavailable | Preserve last valid sentiment, mark freshness degraded. |
| Market data stale | Do not update regime as fresh; publish stale-data event. |
| Classifier fails | Store raw item, publish classification failure, retry or route to DLQ. |
| LLM unavailable | Fall back to deterministic scoring and mark explanation confidence lower. |
| Redis unavailable | Persist to PostgreSQL and skip cache write. |
| Event bus unavailable | Buffer critical events and alert operations. |
| Crisis detection triggers | Prioritize alert, risk notification, and stale-data validation. |

### 7.7 Security and Governance

- Use allowlisted sources only.
- Encrypt source credentials.
- Redact personal data from social text.
- Do not expose raw API keys or tokens in logs.
- Treat external text as untrusted input.
- Protect against prompt injection if LLMs are used.
- Audit source configuration changes.
- Keep intelligence separate from execution authority.
- Require risk validation before any trading action uses intelligence features.

### 7.8 Observability

- Articles collected per source
- Sentiment records collected per platform

- Regime detection latency
- Intelligence snapshot generation latency
- Source error rate
- Source freshness age
- DLQ size
- Cache hit rate
- WebSocket subscriber count

Required alerts:

- News collection stale
- Sentiment collection stale
- Market data stale
- Regime detection failure
