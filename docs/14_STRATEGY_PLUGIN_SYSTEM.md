# Strategy Plugin System Design

This document reviews the current APEX strategy implementation and defines the target Strategy Plugin System for Trend Following, Mean Reversion, Breakout, Scalping, Arbitrage, and User Custom Strategies.

No executable code is generated by this design document.

---

# 0. Current Implementation Review

APEX already has the first layer of a strategy plugin system in place.

Current strengths:

- A shared plugin contract exists in `backend/app/domain/strategies/base.py:9`.
- Built-in plugins already exist for Trend Following, Mean Reversion, Breakout, and Scalping under `backend/plugins/`.
- A registry exists in `backend/app/services/plugin_registry.py:11`.
- A config manager exists in `backend/app/services/strategy_config_manager.py:17`.
- A strategy engine exists in `backend/app/services/strategy_engine.py:12`.
- Backtesting can resolve a plugin from the registry in `backend/app/api/v1/backtest/routes.py:76`.
- Unit tests already cover several built-in plugin behaviors in `backend/tests/test_strategy_plugins.py:11`.

Current limitations:

- Arbitrage is declared as a strategy type in `backend/app/domain/entities/strategy.py:12`, but no arbitrage plugin implementation exists.
- Scalping is implemented as a plugin, but the domain enum in `backend/app/domain/entities/strategy.py:8` does not include it.
- The custom strategy plugin is only a placeholder and returns a pending implementation signal in `backend/plugins/custom/sandbox.py:52`.
- Plugin metadata uses a string `strategy_type` field instead of a strict enum in `backend/app/domain/strategies/types.py:14`.
- The database stores strategy version, but does not store parameter schema history or migration metadata in `backend/app/database/models/trading.py:14`.
- The config manager does not yet load persisted parameters from `StrategyParameter` and currently returns empty defaults in `backend/app/services/strategy_config_manager.py:53`.
- The trading API exposes strategy CRUD, but not plugin discovery, schema, activation, configuration, or sandbox management endpoints in `backend/app/api/v1/trading/routes.py:18`.
- Backtesting currently passes one candle at a time to `analyze()` in `backend/app/services/backtest_engine.py:150`, while many plugins expect a candle history list.

# 2. Plugin Architecture

APEX should keep a modular monolith architecture and make the strategy engine plugin-ready without introducing microservices prematurely.

## 2.1 Target Architecture

The target architecture has six layers:

1. API Layer
2. Strategy Orchestration Layer
3. Plugin Runtime Layer
4. Plugin Implementation Layer
5. Shared Trading Services Layer
6. Data and Infrastructure Layer

## 2.2 Component Responsibilities

| Component | Responsibility |
|---|---|
| Strategy API | Exposes strategy listing, activation, configuration, schema, sandbox upload, and diagnostics. |
| Strategy Engine | Routes normalized market data to active plugins and collects signal results. |
| Plugin Registry | Tracks plugin identity, type, version, status, lifecycle state, and health. |
| Plugin Loader | Discovers built-in plugins and loads approved user custom plugins. |
| Strategy Config Manager | Merges defaults, system overrides, user overrides, and runtime overrides. |
| Plugin Sandbox | Executes user custom strategies with strict resource and import controls. |
| Signal Service | Persists and forwards generated signals to risk and execution services. |
| Risk Service | Validates exposure, leverage, drawdown, concentration, and user limits. |
| Execution Service | Creates live or paper orders after risk approval. |
| Backtest Engine | Replays historical data through the same plugin interface used by live execution. |
| Audit Service | Records plugin activation, configuration changes, sandbox uploads, and signal lineage. |

## 2.3 Runtime Flow

Market data flow:

1. Market collectors produce normalized candles, orderbook, trades, and multi-exchange quotes.
2. Market data is normalized into a strategy-compatible input contract.
3. Strategy Engine selects active plugins by symbol, timeframe, strategy type, and plugin capability.
4. Plugin Registry returns compatible active plugins.
5. Each plugin analyzes the input and returns zero or one signal.
6. Signal Service persists the signal with strategy code, plugin version, and metadata.
7. Risk Service approves or rejects the signal.
8. Execution Service creates orders only for approved signals.

Signal lineage requirement:

- Every signal must include strategy code, plugin version, strategy type, confidence, reason, entry price, stop loss, take profit, input timeframe, symbol, and plugin metadata.

## 2.4 Plugin Categories

| Category | Runtime Trust Level | Execution Location | Dependency Level |
|---|---|---|---|
| Built-in Trend Following | Trusted | Main process | Domain types only |
| Built-in Mean Reversion | Trusted | Main process | Domain types only |
| Built-in Breakout | Trusted | Main process | Domain types only |
| Built-in Scalping | Trusted | Main process | Domain types and normalized orderbook |
| Built-in Arbitrage | Trusted | Main process | Domain types and multi-exchange market data |
| User Custom Strategy | Untrusted | Sandboxed worker or restricted runtime | Whitelisted APIs only |

## 2.5 Recommended Plugin Directory Layout

`backend/plugins/` should remain the built-in plugin boundary.

Recommended layout:

- `trend_following/`
- `mean_reversion/`
- `breakout/`
- `scalping/`
- `arbitrage/`
- `custom/`

Custom user strategies should not be imported directly into the main process. They should be stored as versioned artifacts and executed through the sandbox.

# 3. Strategy Interface

The strategy interface should define a strict contract that every plugin must satisfy.

## 3.1 Required Contract

| Contract Item | Requirement |
|---|---|
| Plugin identity | Unique code, human-readable name, strategy type, semantic version, author, description. |
| Supported markets | Supported asset prefixes or symbols, supported timeframes, required data feeds. |
| Initialization | Initialize once with validated configuration. |
| Validation | Validate configuration before activation. |
| Analysis | Analyze normalized market data and return an optional signal. |
| State | Expose current state for diagnostics and reset state for tests. |
| Schema | Expose JSON Schema for configuration UI and API validation. |
| Lifecycle | Support activate, deactivate, reload, and shutdown semantics. |
| Observability | Emit structured metadata for confidence, reasons, indicators, and errors. |

## 3.2 Required Plugin Methods

| Method | Purpose |
|---|---|
| `metadata` | Returns plugin identity, version, type, author, supported assets, and timeframes. |
| `initialize` | Applies configuration and prepares internal state. |
| `validate_config` | Validates configuration before initialization. |
| `analyze` | Produces a signal from normalized market data. |
| `get_parameters_schema` | Returns the configuration schema. |
| `get_state` | Returns diagnostic state for monitoring and debugging. |
| `reset_state` | Clears internal state, mainly for tests and reloads. |
| `on_market_update` | Optional callback for streaming bar updates. |
| `on_signal_ack` | Optional callback after signal acknowledgement by execution. |

## 3.3 Market Data Contract

All plugins should consume normalized market data instead of exchange-specific objects.

Minimum market data fields:

| Field | Required For |
|---|---|
| symbol | All strategies |
| timeframe | All strategies |
| candles | Trend, Mean Reversion, Breakout, Scalping |
| orderbook | Scalping, Arbitrage |
| trades | Scalping, Arbitrage |
| exchange | Arbitrage |
| bid | Scalping, Arbitrage |
| ask | Scalping, Arbitrage |
| fees | Arbitrage |
| latency_ms | Arbitrage, Scalping |

Candles should include:

- open
- high
- low
- close
- volume
- open_time
- close_time

## 3.4 Signal Contract

Every signal should include:

| Field | Purpose |
|---|---|
| strategy_code | Stable plugin identifier. |
| strategy_version | Plugin semantic version. |
| strategy_type | Trend, Mean Reversion, Breakout, Scalping, Arbitrage, or Custom. |
| signal_type | BUY, SELL, HOLD, or CANCEL depending on future execution needs. |
| confidence | Numeric confidence score from 0 to 100. |
| reason | Human-readable explanation. |
| entry_price | Intended entry price. |
| stop_loss | Risk exit price. |
| take_profit | Target exit price. |
| metadata | Strategy-specific diagnostics and indicator values. |
| generated_at | Signal timestamp. |

## 3.5 Strategy-Type Capability Matrix

| Strategy Type | Required Inputs | Required Configuration | Output Expectations |
|---|---|---|---|
| Trend Following | Candles | Moving average periods, trend confirmation, lookback | BUY on bullish confirmation, SELL on bearish confirmation |
| Mean Reversion | Candles | Bollinger period, RSI period, thresholds | BUY near lower band oversold, SELL near upper band overbought |
| Breakout | Candles and volume | Lookback, threshold, volume multiplier | BUY on resistance breakout, SELL on support breakdown |
| Scalping | Candles and orderbook | Spread threshold, RSI period, profit target, max holding time | Fast micro signals with tight risk limits |
| Arbitrage | Multi-exchange orderbooks, trades, fees, latency | Min spread, max latency, fee model, exchange whitelist | Signal only when net spread survives fees and latency |
| Custom | Defined by user schema | User-defined schema plus sandbox limits | Signal only after sandbox validation and approval |

# 4. Versioning Strategy

APEX should use semantic versioning for every strategy plugin.

## 4.1 Semantic Version Format

Format: `MAJOR.MINOR.PATCH`

| Part | Meaning | Examples |
|---|---|---|
| MAJOR | Breaking interface, incompatible configuration schema, changed signal semantics. | `1.0.0` to `2.0.0` |
| MINOR | Backward-compatible features, new optional parameters, improved signals. | `1.2.0` to `1.3.0` |
| PATCH | Bug fixes, performance improvements, documentation-only logic corrections. | `1.2.3` to `1.2.4` |

## 4.2 Compatibility Rules

| Change Type | Version Impact | Migration Required |
|---|---|---|
| Add optional parameter | MINOR | No |
| Add required parameter | MAJOR | Yes |
| Remove parameter | MAJOR | Yes |
| Change default value | PATCH or MINOR depending on signal impact | Usually no |
| Change signal_type meaning | MAJOR | Yes |
| Change confidence scale | MAJOR | Yes |
| Change required market data | MAJOR | Yes |
| Add optional metadata | MINOR | No |
| Fix bug without signal semantics change | PATCH | No |

## 4.3 Version Storage

The database should store:

| Entity | Data |
|---|---|
| Strategy | code, name, version, strategy_type, status |
| Strategy Parameter Schema | strategy code, schema version, JSON schema, created_at |
| Strategy Parameter Snapshot | strategy code, version, parameter name, value, updated_at |
| Plugin Runtime State | plugin code, version, status, health, last_loaded_at |
| Strategy Migration | from_version, to_version, migration type, status |

## 4.4 Activation Rules

| Scenario | Behavior |
|---|---|
| Plugin version matches active strategy | Activate normally. |
| Plugin version is newer PATCH | Activate normally after validation. |
| Plugin version is newer MINOR | Activate normally if schema migration is backward-compatible. |
| Plugin version is newer MAJOR | Require explicit migration and admin approval. |
| Plugin version is older than active strategy | Do not activate unless rollback is approved. |
| Plugin fails validation | Keep previous active version or keep plugin inactive. |
| Plugin fails health check | Deactivate and alert. |

## 4.5 Hot Reload Policy

Hot reload should be safe and non-disruptive.

Required process:

1. Discover new plugin artifact.
2. Validate plugin identity and interface.
3. Validate configuration schema compatibility.
4. Load new version in a staging state.
5. Run smoke tests using synthetic market data.
6. Drain current in-flight analyses.
7. Swap registry pointer to the new version.
8. Mark old version as inactive but retained for rollback.
9. Roll back automatically if health checks fail.

# 5. Dependency Rules

## 5.1 Dependency Boundary Matrix

| Layer | May Depend On | Must Not Depend On |
|---|---|---|
| Domain | Core types, enums, exceptions | Services, API, database, plugins |
| Plugin Interface | Domain types only | Services, API, database, exchange SDK |
| Built-in Plugins | Domain types and local helper modules | Database, exchange SDK, network, filesystem, secrets |
| Plugin Registry | Plugin interface and metadata | Strategy logic internals |
| Strategy Config Manager | Registry, repositories, schemas | Plugin implementation internals |
| Strategy Engine | Registry, plugin interface, signal types | Concrete plugin classes |
| API | Services, schemas, dependencies | Plugin internals |
| Backtest Engine | Plugin interface, market data, risk helpers | Concrete plugin classes |
| Custom Sandbox | Restricted runtime, schema validator | Main process memory, filesystem, network, secrets |

## 5.2 Plugin Dependency Rules

Required rules:

1. Plugins must declare their plugin code and version.
2. Plugins must declare compatible APEX plugin API version.
3. Plugins must declare required Python version.
4. Plugins must declare required data feeds.
5. Plugins must declare supported assets and timeframes.
6. Plugins must not import `app.services.*` directly.
7. Plugins must not import `app.database.*` directly.
8. Plugins must not import exchange SDKs directly.
9. Plugins must not access environment secrets.
10. Plugins must not perform network calls.
11. Plugins must not read or write arbitrary filesystem paths.
12. Plugins must return normalized signal objects only.
13. Plugins must not mutate global shared state.
14. Plugins must be deterministic for the same input and configuration.

## 5.3 Built-in Plugin Rules

Built-in plugins may use:

- Domain strategy types.
- Local indicator helpers.
- Local parameter definitions.
- Local validation helpers.

Built-in plugins may not use:

- Direct database access.
- Direct exchange API access.
- Direct risk or execution service calls.
- External network calls.
- Hardcoded API keys or secrets.

## 5.4 Custom Strategy Rules

Custom strategies must be treated as untrusted.

Required controls:

- Sandboxed execution.
- Whitelisted imports only.
- No filesystem access.
- No network access.
- No environment variable access.
- No subprocess access.
- No dynamic import of unapproved modules.
- CPU time limit per analysis call.
- Memory limit per execution.
- Execution timeout.
- Audit log for upload, validation, activation, and execution failures.

Allowed custom strategy capabilities:

- Receive normalized market data.
- Receive validated configuration.
- Return normalized signal data.
- Use approved numerical libraries only if available in the sandbox.
- Expose configuration schema.

Disallowed custom strategy capabilities:

- Placing orders directly.
- Reading secrets.
- Calling external APIs.
- Accessing other users' strategies.
- Mutating registry or strategy state outside the sandbox contract.

## 5.5 Service Dependency Rules

Services should depend on abstractions, not concrete plugins.

Recommended dependency direction:

- API depends on services.
- Services depend on domain and repositories.
- Strategy Engine depends on plugin interface.
- Plugin Registry depends on plugin metadata.
- Plugins depend on domain types only.
- Custom Sandbox depends on restricted runtime and validation components.

# 6. Testing Strategy

## 6.1 Test Levels

| Level | Scope | Goal |
|---|---|---|
| Unit Tests | Individual plugins | Validate logic, validation, and signal generation. |
| Contract Tests | Plugin interface | Ensure every plugin satisfies required methods and metadata. |
| Registry Tests | Plugin Registry | Validate discovery, activation, deactivation, routing, and health. |
| Configuration Tests | Strategy Config Manager | Validate defaults, overrides, schema validation, and migration. |
| Sandbox Tests | Custom Sandbox | Validate security isolation and resource limits. |
| Integration Tests | Engine, registry, config, signal service | Validate end-to-end signal generation. |
| Backtest Tests | Historical replay | Validate strategy behavior over known patterns. |
| Performance Tests | Latency and throughput | Validate real-time suitability. |
| Security Tests | Custom strategy and dependency rules | Validate sandbox escape prevention. |
| Regression Tests | Known historical scenarios | Prevent signal behavior drift. |

## 6.2 Required Built-in Strategy Test Scenarios

| Strategy | Required Scenarios |
|---|---|
| Trend Following | Bullish crossover, bearish crossover, insufficient data, sideways market, invalid MA periods. |
| Mean Reversion | Lower band oversold, upper band overbought, normal RSI, insufficient data, invalid RSI thresholds. |
| Breakout | Resistance breakout, support breakdown, low volume rejection, insufficient data, invalid thresholds. |
| Scalping | Tight spread buy, wide spread rejection, overbought sell, insufficient orderbook, invalid profit target. |
| Arbitrage | Positive net spread, negative net spread after fees, stale quote, high latency, missing exchange data. |
| Custom | Valid sandbox strategy, invalid code, timeout, forbidden import, forbidden filesystem access, forbidden network access. |

## 6.3 Contract Test Checklist

Every plugin must pass:

- Has unique plugin code.
- Has semantic version.
- Has valid strategy type.
- Has non-empty name.
- Has description.
- Has supported assets.
- Has supported timeframes.
- Has parameter schema.
- Validates default configuration.
- Rejects invalid configuration.
- Returns `None` when data is insufficient.
- Returns normalized signal when conditions are met.
- Does not access database, network, filesystem, or secrets.
- Does not mutate global state.

## 6.4 Test Data Strategy

Use three categories of test data:

1. Synthetic market data  
   - Known crossovers.
   - Known mean reversion patterns.
   - Known breakout ranges.
   - Known spread opportunities.
   - Known arbitrage opportunities.

2. Historical market data  
   - Stable trending regimes.
   - Range-bound regimes.
   - High volatility regimes.
   - Low liquidity regimes.
   - Exchange outage or stale data regimes.

3. Edge-case data  
   - Missing candles.
   - Zero volume.
   - Negative or zero price.
   - Extreme spread.
   - Duplicate timestamps.
   - Out-of-order candles.
   - Stale orderbook.
   - Missing exchange pair.

## 6.5 Performance Targets

| Area | Target |
|---|---|
| Built-in plugin analysis latency | Less than 100 ms per symbol-timeframe under normal load. |
| Scalping plugin analysis latency | Less than 25 ms where real-time execution is enabled. |
| Arbitrage plugin analysis latency | Less than 10 ms for quote comparison after data normalization. |
| Custom strategy timeout | Configurable, default less than 100 ms per analysis call. |
| Registry lookup | Less than 5 ms for active plugin selection. |
| Configuration validation | Less than 25 ms per update. |

## 6.6 CI/CD Testing Rules

- Run unit tests on every pull request.
- Run contract tests on every plugin change.
- Run integration tests on merge to main.
- Run sandbox security tests before accepting any custom strategy.
- Run performance benchmarks on scheduled builds.
- Fail build if plugin dependency rules are violated.
- Fail build if custom strategy can access filesystem, network, subprocesses, or secrets.

- Add arbitrage plugin structure.

- Add plugin code identity to metadata.
- Add registry health and lifecycle state.
- Add configuration schema persistence.

- All built-in plugins pass contract tests.
- Registry supports lifecycle state.
- Strategy config manager validates persisted parameters.

## Phase 2: Complete Built-in Strategy Coverage

Goals:

- Implement Arbitrage plugin.
- Add multi-exchange market data normalization.
- Add fee-aware profitability calculation.
- Add latency and stale quote handling.
- Add scalping execution guardrails.
- Add trend, breakout, and mean reversion regression scenarios.

Exit criteria:

- All required strategy types have built-in implementations.
- Arbitrage only signals when net profit survives fees and latency.
- Scalping rejects wide spread and stale orderbook data.

## Phase 3: Configuration and Versioning

Goals:

- Add JSON Schema storage for strategy parameters.
- Add parameter snapshot history.
- Add migration metadata.
- Add admin-approved major version upgrade workflow.
- Add rollback support.
- Add runtime configuration API endpoints.

Exit criteria:

- Users can view strategy schema.
- Users can update strategy parameters safely.
- Major version upgrades require explicit approval.
- Rollback to previous active plugin version is possible.

## Phase 4: Custom Strategy Sandbox

Goals:

- Add sandbox runtime.
- Add code validation.
- Add import whitelist.
- Add resource limits.
- Add execution timeout.
- Add audit trail.
- Add custom strategy activation workflow.

Exit criteria:

- Custom strategies cannot access filesystem, network, subprocesses, or secrets.
- Custom strategies can receive normalized data and return normalized signals.
- Sandbox failures are isolated and logged.

## Phase 5: Observability and Risk Integration

Goals:

- Add plugin health metrics.
- Add signal lineage tracking.
- Add strategy performance dashboards.
- Add risk-aware plugin activation.
- Add automatic deactivation on repeated failures.
- Add audit logs for configuration and sandbox changes.

Exit criteria:

- Operators can see active plugin version, health, latency, and failure rate.
- Risk service can block strategies that exceed configured limits.
- Signals include full strategy and plugin lineage.

## Phase 6: External Plugin Distribution

Goals:

- Add plugin packaging rules.
- Add plugin manifest requirements.
- Add signed plugin artifacts.
- Add plugin marketplace or internal plugin catalog.
- Add compatibility checks before installation.
- Add staged rollout by user or environment.

Exit criteria:

- External plugins can be installed only after validation and approval.
- Plugin compatibility is checked before activation.
- Broken external plugins cannot crash the main process.

## Phase 7: Advanced Strategy Runtime

Goals:

- Add multi-strategy portfolio allocation.
- Add strategy conflict resolution.
- Add ensemble signals.
- Add online learning feedback loops.
- Add per-strategy risk budgets.
